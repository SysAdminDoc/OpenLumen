# v0.5.0 → F-Droid Release Readiness Checklist

> Single-page, copy-paste-friendly checklist for cutting v0.5.0 and
> submitting it to F-Droid as the **first** release on the F-Droid
> store. Maps every Now-tier ROADMAP item to a specific command or
> file edit.
>
> Tied to ROADMAP rev 4.1 (2026-05-17) and the F-Droid submission
> recipe at [fdroid-recipe-draft.md](fdroid-recipe-draft.md). Read
> [release-checklist.md](release-checklist.md) for the per-release
> mechanical steps that apply to every cut; this document layers the
> v0.5.0-specific gates on top.

## Hard gates (all must pass)

### Gate 1 — Audit hardening fixes are landed

The 2026-05-17 in-tree audit pass touched 15 Kotlin / test / strings
files and is now committed on the local `main` history after v0.4.0
(`f1c55c8` plus later roadmap/release-readiness documentation commits).
Before cutting v0.5.0, verify the tree is clean and rerun the release
checks in [release-checklist.md](release-checklist.md).

The CHANGELOG `[Unreleased]` "Hardening" subsection (added in rev 4.1)
documents the line-item list.

### Gate 2 — Pass-2 correctness fixes (C132-C136)

Status: **landed 2026-05-17 after rev 5**.

- **C132** — `LumenService.applyMatrix` ramp-scheduling atomicity.
  Implemented with a dedicated `rampMutex` around transition cancel/join,
  `lastApplied` reads, and new ramp launch.
- **C133** — `LumenService.clearAndStop` cancel-and-join
  `transitionJob` before `engine?.clear()`.
- **C134** — `ColorDisplayManagerEngine.load` reset `cdm = null` when
  returning null after a partial reflective failure.
- **C135** — `OverlayEngine.installView` thread-safety with
  `apply`/`clear`, implemented with an internal `viewLock` around view /
  `WindowManager` mutation.
- **C136** — SurfaceFlinger / KCAL apply-clear exit checking now
  invalidates stale cached transaction codes or sysfs paths on failure.

Validation from the landing pass:

```bash
./gradlew :app:testDebugUnitTest :core-engine:test :core-schedule:test :core-prefs:test --rerun-tasks --stacktrace
./gradlew :app:assembleDebug --stacktrace
./gradlew :app:lintDebug --stacktrace
git diff --check
```

### Gate 5 — F-Droid recipe + buildserver dry run

Refer to [fdroid-recipe-draft.md](fdroid-recipe-draft.md) for the
exact YAML to commit to `fdroiddata`. Before the MR:

- Install F-Droid server tools locally:

  ```bash
  pipx install fdroidserver
  fdroid lint com.openlumen          # syntax check on the recipe
  fdroid build com.openlumen --on-server   # optional local build dry run
  ```

- Apply the `version-control-info.textproto` reproducibility fix
  (C120) per [reproducible-build.md](reproducible-build.md) and the
  recipe at https://forum.f-droid.org/t/how-can-i-prevent-version-control-info-textproto-from-being-included-in-my-apk/33196.

### Gate 6 — Versioning + release tag

In `app/build.gradle.kts`:

```kotlin
versionCode = 6
versionName = "0.5.0"
```

Then:

```bash
# Update fastlane changelog for the new version
mv fastlane/metadata/android/en-US/changelogs/{5,6}.txt
# Edit 6.txt to match v0.5.0 (≤500 chars)

git commit -am "Bump version to 0.5.0 (versionCode 6)"
git tag v0.5.0
git push origin main v0.5.0
```

The release workflow (`workflow_dispatch` on `.github/workflows/release.yml`)
takes the version as an input, builds, attests, hashes, and uploads to a
GitHub Release.

### Gate 7 — CI green on the release commit

All three workflows must succeed:

- `ci.yml` — `assembleDebug + lint`, module unit tests,
  **`permissions-audit`** (network + location + phone + usage + a11y
  blocklist as of rev 4.1).
- `release.yml` — release APK + no-banned-permissions assertion +
  SHA-256 + build-provenance attestation.
- `sbom.yml` — SPDX SBOM + Anchore advisory scan; non-blocking, but
  any new High / Critical findings must be triaged before merging the
  release commit.

### Gate 8 — F-Droid submission MR (C140)

Once Gates 1–7 pass:

1. Fork `gitlab.com/fdroid/fdroiddata` to your GitLab account.
2. Add `metadata/com.openlumen.yml` per
   [fdroid-recipe-draft.md](fdroid-recipe-draft.md).
3. Open MR titled `New app: com.openlumen (OpenLumen)`.
4. Wait 24–48h post-merge for the repo to refresh.

## Pre-flight commands (run all of these on `main` before tagging)

```bash
cd Z:/repos/OpenLumen

# 1. Build artefacts
./gradlew clean
./gradlew :app:assembleDebug :app:lint --stacktrace

# 2. Tests
./gradlew \
  :app:testDebugUnitTest \
  :core-engine:test \
  :core-schedule:test \
  :core-prefs:test \
  --stacktrace

# 3. Manifest + classpath audit (matches the permissions-audit CI job)
./gradlew :app:assembleDebug
APK=$(ls app/build/outputs/apk/debug/app-debug.apk)
AAPT=$ANDROID_HOME/build-tools/$(ls -1 $ANDROID_HOME/build-tools | tail -n1)/aapt2
"$AAPT" dump permissions "$APK" | \
  grep -E "android\\.permission\\.(INTERNET|ACCESS_NETWORK_STATE|ACCESS_WIFI_STATE|ACCESS_FINE_LOCATION|ACCESS_COARSE_LOCATION|ACCESS_BACKGROUND_LOCATION|READ_PHONE_STATE|QUERY_ALL_PACKAGES|PACKAGE_USAGE_STATS|BIND_ACCESSIBILITY_SERVICE)" \
  && echo "FAIL: banned permission in merged manifest" || echo "OK"

./gradlew :app:dependencies --configuration releaseRuntimeClasspath > /tmp/deps.txt
grep -E "(play-services|firebase|com\\.google\\.android\\.gms)" /tmp/deps.txt \
  && echo "FAIL: disallowed dependency" || echo "OK"

# 4. Release APK (requires keystore env vars)
./gradlew :app:assembleRelease --stacktrace
SRC=$(ls app/build/outputs/apk/release/app-release*.apk | head -n 1)
"$AAPT" dump permissions "$SRC" | grep INTERNET && echo "FAIL" || echo "OK"
sha256sum "$SRC"
```

- The git push policy. Pushes to `SysAdminDoc/OpenLumen` from this VM
  403; the maintainer pushes from elsewhere.

  v0.4.0 ships first or whether to wait for v0.5.0.
